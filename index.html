第11章 序列化

11.1 谨慎的实现Serializable接口

实现Serializable借接口有如下代价：

  • 一旦一个类被发布，就大大降低了“改变这个类的实现”的灵活性；
  • 增加了出现BUG和漏洞的可能性；
  • 随着类发行新的版本，相关的测试负担也增加了；

为了继承而设计的类或者是用户接口，尽量避免使用Serializable接口；

11.2 考虑使用自定义的序列化形式

理想的序列化形式应该只包含该对象所表示的逻辑数据，而逻辑数据和物理表示法应该是各自独立的。

transient修饰符表明这个实例域将从一个类的默认序列化形式中省略掉。

如果在读取整个对象状态的任何其他方法上强制任何同步，则也必须在对象序列化上强制这种同步。

11.3 保护性的编写readObject方法

保护性的编写readObject方法。readObject方法就相当于一个公有的构造器，相当于提供了一种从输入流初始化对象的途径。通过构造特殊的包含“恶意编制的对象引用”的字节流，可以访问到对象的不可变域，破坏类原有的约束条件和安全性。

编写更加健壮的readObject方法的指导方针：

  • 对于对象引用域必须保持为私有的类，要保护性的拷贝这些域中的每个对象。不可变类的可变组件就属于这一类别。
  • 对于任何约束条件，如果检查失败，则抛出一个InvalidObjectException异常，这些检查应该跟在所有的保护性拷贝之后。
  • 如果整个对象在被反序列化以后必须进行验证，就应该使用ObjectInputValidation接口;
  • 无论是直接还是间接方式，都不要调用类中任何可被覆盖的方法。

11.4 对于实例控制，枚举类型优先于readResolve

对象在反序列化以后，新建对象上面的readResolve方法就会被调用。然后，该方法返回的对象引用将被返回，取代新建的对象。

如果以来readRosolve进行实例控制，带有对象引用类型的所有实例域则都必须申明为transient的。

如果将一个可序列化的实例受控的类编写成枚举。就可以绝对保证除了所申明的常量之外，不会有别的实例。

11.5 考虑用序列化代理代替序列化实例

序列化模式相当简单。首先，为可序列化的类设计一个私有的静态嵌套类，精确的表示外围类的实例的逻辑状态。这个嵌套类被称作序列化代理。他应该有一个单独的构造器，其参数类型就是那个外围类，这个构造器只从他的参数中复制数据，他不需要进行任何一致性检查和保护性拷贝。从设计的角度来看，序列化代理的默认序列化形式是外围类最好的序列化形式。外围类和其序列代理都必须申请实现Serializeable接口。

4.9 用函数对象表示策略

函数指针的主要作用就是实现策略（Strategy）模式，为了在Java中实现这种事，要申明一个接口来表示该策略，并且为每个具体策略申明一个实现了该接口的类。当一个具体策略只被使用一次时，通常使用匿名类来申明和实例化这个具体策略类。当一个具体策略是被设计用来重复使用的，它的类通常就要被实现为私有的静态成员类，并通过公有的静态final域被导出，其类型为该策略接口。

4.10 优先考虑静态成员类

嵌套类主要包含四种：

静态成员类的一种常见用法就是作为提供有的辅助类，仅当与它的外部类一起使用时才有意义。

非静态成员类的一种常见用法是定义一个Adapter,它允许外部类的实例被看做是另一个不相关的的类的实例。

私有静态成员类的一种常见用法是用来代表外围类所代表的对象的组件。

匿名类的三种常见用法：

  • 动态创建函数对象，比如匿名Comparator实例；
  • 创建过程对象对象，不如Runable、Thread和TimeTask实例；
  • 用在静态工厂方法的内部

四种嵌套类，如果一个嵌套类需要在单个方法之外任然是可见的，或者它太长了不适合放在方法内部，就应该使用成员类。如果成员类的每个实例都需要一个指向其外围实例的引用，就要把成员类当做非静态的，否则就做成静态的。假设这个嵌套类属于一个方法的内部，如果只需要在一个地方创建实例并且已经有了一个预置的类型可以说明这个类的特征，就把它做成内部类，否则就选择局部类。

  • 平行迭代—>如果需要并行遍历多个集合，就需要显示的控制迭代器或者索引变量，以便他们可以得到同步前移；

Effective Java 中文版（第二版）第9章 异常

第9章 异常

9.1 只针对异常的情况下才使用异常

异常只用于异常的情况下，它们永远不应该用于正常的的控制流。

9.2 对可恢复的情况使用受检异常，对编程错误使用运行时异常

Java包含三种可抛出结构：受检的异常(checked exception),运行时异常(run-time exception)和错误(error)。

如果调用者能够适当的恢复，使用受检异常。

用运行时异常来表示编程错误。

所有未受检的抛出异常都应该是RuntimeException的子类。

9.3 避免不必要的使用受检的异常

9.4 优先使用标准的异常

9.5 抛出与抽象相对应的异常

更高层的实现应该捕获底层的异常，同时抛出可以按照高层抽象进行解释的异常。

异常转译与不加选择的从底层传递异常的做法相比有所改进，但是它也不能被滥用。

9.6 每个方法抛出的异常都要有文档

可以使用@throws标签记录下一个方法可能抛出的每个未受检异常，但是不要用throws关键字将未受检的异常包含在方法的申明中。

要为每个受检异常提供单独的thorws字句，不要为未受检的异常提供throws语句。

9.7 在细节消包含能捕获失败的信息

异常的细节消息应该捕获住失败，为了捕获失败，异常的细节信息应该包含所有“对该异常有贡献”的参数和域的值。

9.8 努力使失败保持原子性

失败的方法调用应该使对象保持在被调用之前的状态。具有这种属性的方法被称为失败原子性。实现这种效果有以下途径：

  • 设计一个不可变对象；
  • 可变对象在操作之前检查参数的合法性；
  • 调整计算处理过程的顺序，使得任何可能会导致失败的计算部分都在对象状态被修改之前发生；
  • 采用一段恢复代码拦截操作过程中发生的失败，并使对象回滚到操作之前的状态上；
  • 在对象的临时拷贝上面执行操作；

9.9 不要忽略异常

空的catch块不能使用，至少要打印出相关的日志记录。

  • 把参数转换成正确的类型
  • 对于该类中的每个关键域，检查参数中的域是否与该对象的对应
